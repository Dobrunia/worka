# Worka — Проект тайм-трекера для Windows 10

## Обзор проекта

**Worka** — это desktop-приложение для автоматического трекинга активности пользователя на Windows 10. Приложение работает в системном трее, периодически собирает данные об активном приложении, статусе idle и количестве ввода (клавиатура/мышь), сохраняет их в локальную SQLite-базу и предоставляет дашборд с аналитикой.

### Основные характеристики
- **Платформа:** Windows 10
- **Тип:** Tray-приложение с UI-дашбордом
- **Конфиденциальность:** Локальное хранение данных, трекинг только метаданных (без содержания текста)

## Технологический стек

| Компонент | Технология |
|-----------|------------|
| Desktop shell | Tauri v2 |
| UI | Vue 3 + Vite + TypeScript |
| UI библиотека | dobruniaui-vue |
| Графики | chart.js, vue-chartjs |
| Тестирование UI | vitest |
| Backend (внутри Tauri) | Rust |
| Windows API | windows crate (v0.58) |
| База данных | SQLite (rusqlite v0.38) |

### Причины выбора
- **Tauri:** малый размер, хорошая интеграция с Windows, поддержка tray, системная логика на Rust
- **SQLite:** локальная, простая, быстрая, не требует сервера

```

│                     npm run dev (shell)                     │
│                          PID: 12345                         │
└─────────────────────┬───────────────────────────────────────┘
                      │ запускает
                      ▼
┌─────────────────────────────────────────────────────────────┐
│                    tauri dev (Tauri CLI)                    │
│                          PID: 12346                         │
└──────────────┬────────────────────────────────┬─────────────┘
               │                                │
               │ запускает                      │ запускает
               ▼                                ▼
    ┌──────────────────────┐      ┌──────────────────────────┐
    │   npm run dev:vite   │      │   cargo run --bin worka  │
    │   (Vite dev server)  │      │   (Tauri Rust app)       │
    │   PID: 12347         │      │   PID: 12348             │
    │   Порт: 1420         │      │   + Windows tray         │
    └──────────────────────┘      │   + SQLite connection    │
                                  │   + Sampler thread       │
                                  └──────────────────────────┘
```

### Компоненты и их жизненный цикл

| Компонент | Процесс | Зависимости | Жизненный цикл |
|-----------|---------|-------------|----------------|
| **npm run dev** | Shell | — | Родительский процесс, закрывается по Ctrl+C |
| **tauri dev** | Tauri CLI | npm | Управляет Vite и Rust, закрывается по Ctrl+C |
| **Vite dev server** | Node.js | tauri dev | Frontend dev server, порт 1420 |
| **worka.exe** | Rust | tauri dev | Backend + UI окно + трей + сэмплер |

### Важные правила

1. **Vite может работать без Rust** — для разработки UI (`npm run dev:vite`)
2. **Rust НЕ должен работать без Vite** — будет белый экран
3. **При закрытии Ctrl+C** — все процессы должны закрыться
4. **При пересборке** — старый процесс закрывается, новый запускается

### Single-instance защита

- **Windows Mutex** — `Local\WorkaSingleInstanceMutex`
- **Второй процесс закрывается сразу** — `std::process::exit(0)`
- **Только одна иконка в трее** — гарантировано mutex

### Компоненты

#### 1. Agent (фоновый сборщик)
Работает в фоне, пока приложение в трее.

- **Канал 1 — Время:** Сэмплинг каждые N секунд (активное приложение, статус idle, опционально заголовок окна)
- **Канал 2 — Ввод:** Счётчик нажатий клавиатуры и кликов мыши через глобальные хуки Windows (только количество)
- **Канал 3 — Метаданные:** Состояние паузы, настройки интервала, порог idle, включён ли трекинг

#### 2. Storage (SQLite)
- Миграции схемы
- Запись сэмплов (activity_sample, input_sample)
- Агрегированные запросы для дашборда

#### 3. Dashboard (Vue UI)
4 основных экрана:
- **Сегодня:** KPI, топ приложений, график интенсивности ввода
- **Неделя:** Активное/idle по дням, топ за неделю, суммарный ввод
- **Таймлайн:** Лента дня (приложение/idle), опционально дорожка ввода
- **Настройки:** Интервал опроса, порог idle, переключатели приватности, пауза/продолжить, автозапуск

### Сущности данных

| Таблица | Описание |
|---------|----------|
| `app` | Справочник приложений (id, exe_path, display_name) |
| `activity_sample` | Снимок состояния по времени (timestamp, app_id, is_idle, window_title) |
| `input_sample` | Агрегат ввода по времени (timestamp_bucket, keyboard_presses, mouse_clicks, mouse_wheel) |
| `settings` | Настройки (sample_interval_seconds, idle_threshold_seconds, track_window_titles, track_input, autostart, paused) |

## Функциональные уровни

### Уровень 1 — MVP (текущая разработка)
- Работа в трее, выход только через меню трея

- Локальный API
- Синхронизация между устройствами
- Плагинная система

## Поведение приложения

### Трей (системный трей Windows)

**Левый клик по иконке в трее:**
- Открывается контекстное меню с пунктами:
  - **Открыть** — показать окно дашборда
  - **Выйти** — полностью закрыть приложение (сбор данных останавливается)

**Закрытие окна (крестик):**
- Окно скрывается в трей (не закрывается)
- Сбор данных продолжается в фоне

**Single-instance:**
- При запуске второго экземпляра открывается окно первого
- Используется `tauri-plugin-single-instance`
- Флаг `AtomicBool` для остановки сэмплера при выходе

**Сбор данных:**
- Периодический сэмплер работает каждые 10 секунд
- При выходе сэмплер останавливается через флаг
- `.db` файлы игнорируются при пересборке (dev режим)

## Правила для AI-помощника

### Роль
Помощник по документации проекта. Поддерживает актуальность 3 файлов:
- `functional.md` (требования и уровни фич)
- `stack.md` (технологический стек)
- `architecture.md` (архитектура, компоненты, сущности, экраны, поведение)

### Главный принцип
`architecture.md` должен быть согласован с `functional.md` и `stack.md`. При изменениях в требованиях или стеке — обновлять `architecture.md`.

### Workflow на каждое изменение
1. Прочитать текущие версии `functional.md`, `stack.md`, `architecture.md`
2. Определить, что поменялось (требования / стек / UX / данные)
3. Сформулировать "impact list": какие разделы архитектуры затронуты
4. Обновить `architecture.md` так, чтобы компоненты, сущности и экраны соответствовали требованиям
5. **Не менять** `functional.md` и `stack.md` без явного запроса

### Правила согласованности
- Нельзя добавлять новые фичи в архитектуру, если их нет в `functional.md` (кроме пометок "опционально/уровень 2/уровень 3")
- Ввод (клавиатура/мышь) всегда описывается как "только количество", без сохранения текста
- Архитектура ориентирована на Windows 10

### Стиль `architecture.md`
- Короткие чёткие заголовки
- Списки вместо полотен
- Разделы: схема, компоненты, сущности данных, экраны UI, поведение приложения
- Одинаковые названия сущностей везде (activity_sample, input_sample)
- При новом поле/таблице — объяснение зачем (1 строкой)

### Формат ответа при обновлении
- В начале: 3–6 буллетов — что изменил(а) и почему
- Затем: ПОЛНАЯ новая версия `architecture.md` целиком (не дифф)

### Правило минимальных изменений
Менять только нужные разделы `architecture.md`, сохраняя общую структуру. Не переименовывать сущности без необходимости.

## Frontend архитектура

### Компоненты

**View компоненты** (`src/views/`) — экраны приложения:
- `TodayView.vue` — дашборд за сегодня
- `WeekView.vue` — статистика за неделю
- `TimelineView.vue` — детальный таймлайн дня
- `SettingsView.vue` — настройки приложения

**Layout компоненты** (`src/components/layout/`):
- `AppHeader.vue` — шапка с логотипом и статусом
- `AppNavigation.vue` — навигация по вкладкам
- `AppFooter.vue` — подвал с версией

**UI компоненты** (`src/components/ui/`):
- `KpiCard.vue` — карточка KPI (label + value)
- `TopAppsList.vue` — список топ приложений с прогресс-барами

### Алиасы

В проекте настроен алиас `@` для импорта из `src/`:

```typescript
// ✅ Правильно
import KpiCard from "@/components/ui/KpiCard.vue";

// ❌ Неправильно
import KpiCard from "../../components/ui/KpiCard.vue";
```

Настройка в `tsconfig.json`:
```json
{
  "paths": {
    "@/*": ["src/*"]
  }
}
```

Приложение предоставляет следующие Rust-команды для вызова из frontend:

| Команда | Описание | Возвращает |
|---------|----------|------------|
| `get_settings()` | Получить текущие настройки | `AppState` |
| `set_settings(...)` | Обновить настройки | `()` |
| `sample_activity()` | Сделать сэмпл активности | `Option<ActivitySample>` |
| `get_today_summary()` | Получить сводку за сегодня | `JSON` |
| `get_week_summary()` | Получить сводку за неделю | `JSON` |
| `get_timeline(date)` | Получить таймлайн за дату | `JSON` |
| `toggle_pause()` | Переключить паузу | `bool` |

### Пример вызова из Vue:

```typescript
import { invoke } from "@tauri-apps/api/core";

// Получить настройки
const settings = await invoke("get_settings");

// Переключить паузу
const isPaused = await invoke("toggle_pause");
```
